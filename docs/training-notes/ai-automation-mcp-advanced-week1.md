## 一、AI Coding 基础

### 1.1 什么是 AI Coding

AI Coding 是指利用大语言模型（LLM）辅助软件开发的实践，覆盖需求分析、代码生成、重构、调试和文档整理等环节。它不是简单的代码补全，而是能够理解项目上下文、拆解任务并生成完整代码片段的工作方式。

### 1.2 AI Coding vs 普通代码补全

| 对比维度 | 普通代码补全 | AI Coding |
|----------|-------------|-----------|
| 工作方式 | 基于模式匹配，补全当前行 | 基于语义理解，生成完整功能 |
| 上下文 | 当前文件和光标位置 | 项目结构、多文件、对话历史 |
| 能力范围 | 变量名、函数名建议 | 组件、接口、配置文件生成 |
| 典型工具 | IDE 内置补全 | Claude Code、GitHub Copilot |
| 理解能力 | 无 | 能理解自然语言需求描述 |

### 1.3 AI 能做什么、不能做什么

**能做的：**

- 根据需求描述生成组件和页面
- 重构代码、提取公共逻辑
- 分析和解释错误信息
- 生成测试用例
- 整理项目文档和注释
- 检查代码规范和潜在问题

**不能做的：**

- 替代开发者做出架构决策
- 保证生成的代码零缺陷
- 自动了解项目全部业务逻辑
- 在生产环境中安全地自主操作

> **关键原则**：AI 生成的代码必须经过人工阅读和验证。AI 是辅助工具，不是替代品。

## 二、如何写清楚任务指令

### 2.1 完整指令的要素

一条较完整的开发指令应包含以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **背景** | 当前项目情况和已有功能 | "博客已有文章列表和详情页" |
| **目标** | 要实现什么功能 | "增加文章收藏功能" |
| **文件范围** | 需要修改哪些文件 | "修改 ArticlePage.jsx 和 storage.js" |
| **数据结构** | 数据如何存储 | "收藏数据存入 localStorage，格式为 { userId, articleId }" |
| **UI 要求** | 界面风格和交互 | "按钮使用品牌色，收藏后图标变为实心" |
| **限制条件** | 不能修改什么 | "不要修改 ArticleCard 组件" |
| **验收标准** | 如何判断完成 | "未登录用户点击收藏跳转登录页" |

### 2.2 模糊指令 vs 清晰指令

**❌ 模糊指令：**

```text
帮我增加评论功能。
```

**✅ 较清晰的指令：**

```text
请在现有评论区增加评论编辑功能。
要求：
- 仅评论作者可以编辑自己的评论
- 编辑时显示原内容，保留原创建时间
- 保存后更新 updatedAt 字段
- 使用 localStorage 存储
- 兼容旧评论数据（无 updatedAt 的评论不显示编辑时间）
- 不要修改 CommentSection.css
```

### 2.3 指令模板

```text
任务背景 → 当前项目状态
实现目标 → 要达成的效果
具体要求 → 功能、数据、UI 细节
文件范围 → 需要修改的文件列表
限制条件 → 禁止修改的内容
验收标准 → 如何判断完成
```

## 三、上下文管理

### 3.1 AI 为什么需要上下文

AI 不是一个了解你项目的独立个体。每次对话开始时，它对项目一无所知。通过提供上下文——项目目录结构、现有组件、数据流、错误信息——才能让它产出准确的代码。

### 3.2 上下文原则

| 原则 | 说明 |
|------|------|
| **提供相关文件** | 让 AI 先读取要修改的文件，而不是凭记忆描述 |
| **提供真实报错** | 粘贴完整报错比说"运行不了"有效得多 |
| **限制范围** | 指定具体文件，比让 AI 自由修改整个项目更安全 |
| **分层提供** | 先给结构，再给细节，不要一次塞入大量无关内容 |
| **确认来源** | 修改前先确认数据来源和组件关系 |

### 3.3 长任务的分段策略

对于复杂任务，分成多个阶段执行：

```text
阶段 1：分析 — 阅读相关文件，理解现有结构
阶段 2：实现 — 逐步修改代码，每次只改 1-2 个文件
阶段 3：检查 — 构建验证、功能测试
阶段 4：总结 — 记录修改内容和注意事项
```

## 四、AI 自动化与工作流

### 4.1 什么是 AI 自动化

AI 自动化是将重复性的开发任务交给 AI 执行的过程。与传统脚本自动化不同，AI 自动化能够处理非结构化输入（如自然语言描述、错误日志）并灵活调整执行方式。

### 4.2 AI 自动化 vs 脚本自动化

| 对比 | 脚本自动化 | AI 自动化 |
|------|-----------|-----------|
| 输入 | 固定格式 | 自然语言、非结构化数据 |
| 灵活性 | 低，需要预定义所有路径 | 高，可根据上下文调整 |
| 适用场景 | 构建、部署、定时任务 | 代码审查、文档生成、问题排查 |
| 可预测性 | 高，输出确定 | 较低，需要人工验证 |

### 4.3 AI 自动化的常见场景

结合项目实际：

- 根据模板自动生成学习笔记
- 检查 Markdown 格式和链接有效性
- 根据数据自动创建博客文章
- 检查和更新文章分类与标签
- 扫描代码中的常见问题
- 生成测试用例骨架
- 根据变更自动整理提交说明
- 分析错误日志并提供排查方向

### 4.4 基本工作流程

```text
接收任务
  → 读取项目上下文（文件、配置、数据）
    → 拆分步骤（分析 → 实现 → 验证）
      → 调用工具（读写文件、执行命令）
        → 修改文件
          → 执行检查（构建、测试、格式）
            → 输出结果
```

## 五、Agent、Tool 与 Workflow

### 5.1 三个核心概念

**Agent**

能够根据目标自主分析、规划和执行任务的 AI 系统。Agent 会自己决定"下一步做什么"——读取哪个文件、调用哪个工具、如何拆分任务。

**Tool**

AI 可以调用的具体执行能力。Tool 是 Agent 的"手和眼睛"：

- 读取文件内容
- 修改代码
- 执行终端命令
- 查询数据库
- 调用外部 API
- 获取 Git 信息

**Workflow**

预先定义好的执行流程，步骤固定、结果更容易预测和复现。适合"每次都要这样做"的重复任务。

### 5.2 三者对比

| 概念 | 核心作用 | 灵活性 | 示例 |
|------|----------|--------|------|
| **Agent** | 根据目标自主选择步骤 | 高 | "排查项目启动失败的原因" |
| **Tool** | 提供具体执行能力 | — | 读取文件、执行命令、搜索代码 |
| **Workflow** | 按固定步骤执行任务 | 低 | 代码检查 → 构建 → 测试 → 发布 |

### 5.3 如何选择

- **Agent** — 任务目标明确但路径不确定时使用。需要更强的权限控制。
- **Workflow** — 步骤固定、需要频繁执行的重复任务。结果稳定可预期。
- **Tool** — 两者都依赖 Tool 来执行具体操作，Tool 是基础设施。

## 六、MCP 基础

### 6.1 MCP 是什么

MCP（Model Context Protocol）是一个开放协议，用于让 AI 模型以统一的方式连接外部工具和数据源。它定义了 AI 客户端（Client）与工具提供方（Server）之间的通信标准。

### 6.2 MCP 不是...

| 容易误解 | 实际情况 |
|----------|----------|
| ~~MCP 是一个 AI 模型~~ | MCP 是连接协议，不是模型 |
| ~~MCP 是数据库~~ | MCP 可以连接数据库，但本身不存储数据 |
| ~~MCP 只能连接文件系统~~ | MCP 可以连接数据库、Git、浏览器、API 等 |

### 6.3 MCP 的工作方式

```text
用户提出任务："帮我查看最近提交的代码"
  → AI 客户端（如 Claude Code）理解任务
    → 客户端通过 MCP 调用 Git Server 提供的 get_commits 工具
      → MCP Server 执行 git log，返回提交记录
        → 客户端分析结果，整理为可读的摘要
          → 返回给用户
```

### 6.4 MCP 的核心角色

| 角色 | 职责 | 示例 |
|------|------|------|
| **MCP Client** | 发起请求的一方（AI 应用） | Claude Code、Claude Desktop |
| **MCP Server** | 提供工具和数据的一方 | 文件系统 Server、Git Server、数据库 Server |

> MCP 的价值在于"统一"——不管是连接文件还是连接数据库，AI 客户端都使用同一套协议。开发者不需要为每种工具写不同的对接代码。

### 6.5 使用 MCP 的安全原则

- 检查 MCP Server 的来源和权限范围
- 读取权限和写入权限应分开管理
- 高风险操作（删除、部署）应保留人工确认
- 不要默认信任第三方 MCP Server

## 七、MCP 的常见应用场景

### 7.1 软件开发相关场景

| 场景 | 说明 | 典型工具 |
|------|------|----------|
| 读取项目文件 | AI 直接访问代码仓库 | 文件系统 MCP Server |
| 查询 Git 历史 | 查看提交记录、diff、blame | Git MCP Server |
| 查询数据库 | 读取但禁止修改生产数据 | 数据库 MCP Server |
| 读取团队文档 | 查阅 Wiki、设计文档 | 文档平台 MCP Server |
| 页面检查 | 浏览器截图、控制台日志 | 浏览器 MCP Server |
| 任务管理 | 查询 Issue、更新状态 | 项目管理 MCP Server |

### 7.2 以博客项目为例

```text
任务：检查所有文章的分类是否正确

AI 执行流程：
  读取 storage.js → 找到 CATEGORIES 数组
    → 遍历所有文章 → 检查 category 字段
      → 找到缺少 category 的文章 → 补充默认分类
        → 检查分类页面是否正常显示
```

### 7.3 权限分级建议

```text
读取文件 < 修改文件 < 执行命令 < 部署上线
```

权限和风险逐级提高。每提升一级，都需要更强的验证和更明确的授权。

## 八、AI 自动化中的安全问题

### 8.1 绝不能做的事

| 禁止行为 | 原因 |
|----------|------|
| 向 AI 提供密码、Token、私钥 | 对话内容可能被记录 |
| 让 AI 直接操作生产数据库 | 一条错误 SQL 可能造成数据丢失 |
| 自动执行来源不明的命令 | 可能是恶意指令 |
| 默认信任第三方 MCP Server | 未知来源的工具可能有风险 |

### 8.2 应该养成的习惯

- ✅ 修改前查看 `git status`
- ✅ 在独立分支上让 AI 操作
- ✅ 提交前检查 `git diff`
- ✅ 运行测试验证修改
- ✅ 不提交包含敏感信息的文件
- ✅ 删除、覆盖和部署操作保留人工确认
- ✅ AI 生成的依赖名称和命令需要核实

### 8.3 安全层级

```text
读取文件（低风险）
  → 修改文件（中风险，需检查 diff）
    → 执行命令（高风险，需确认）
      → 部署上线（极高风险，需审核流程）
```

## 九、AI 辅助调试

### 9.1 推荐调试流程

```text
复现问题 → 确认现象
  → 获取完整报错（复制粘贴，不要转述）
    → 找到相关文件（指定具体文件路径）
      → 分析数据流和调用关系
        → 提出可能原因（逐一排查）
          → 小范围修改（每次只改 1-2 处）
            → 重新运行验证
              → 记录修复过程
```

### 9.2 常见无效做法

| 无效做法 | 为什么无效 |
|----------|-----------|
| 只说"页面坏了" | AI 无法定位问题 |
| 一次修改大量文件 | 难以确定哪个改动生效 |
| 没看报错就重复生成代码 | 错误原因没有被分析 |
| 修改后不运行检查 | 无法确认问题是否解决 |
| 为一个问题破坏其他功能 | 缺乏回归验证 |

### 9.3 实际调试示例

**问题：React 页面没有更新**

```text
排查步骤：
1. 检查 state 是否正确更新（console.log）
2. 检查 useEffect 依赖数组是否完整
3. 检查 props 是否传递正确
4. 检查 localStorage 数据格式是否匹配
```

**问题：API 返回 500**

```text
排查步骤：
1. 查看后端日志中的完整报错
2. 检查请求参数格式
3. 检查数据库连接
4. 检查路由和处理函数是否匹配
```

## 十、AI 辅助小游戏开发

### 10.1 AI 在游戏开发中的角色

AI 可以帮助完成小游戏开发中的许多环节：

- 生成基础游戏逻辑（循环、状态、计分）
- 拆分游戏状态（等待、进行、结束）
- 生成 UI 组件（开始按钮、分数显示）
- 添加事件处理（键盘、鼠标、触摸）
- 编写碰撞检测逻辑
- 调整游戏参数（速度、难度）
- 生成 README 和操作说明

### 10.2 小游戏涉及的基础知识

| 知识点 | 说明 | 示例 |
|--------|------|------|
| **游戏循环** | 持续更新和渲染的循环 | `requestAnimationFrame` 或 `setInterval` |
| **状态管理** | 追踪游戏当前状态 | `useState` 管理 waiting/playing/finished |
| **用户输入** | 响应用户操作 | `onClick`、`onKeyDown` |
| **随机数** | 增加不可预测性 | `Math.random()` 控制出现时机 |
| **碰撞判断** | 检测两个对象是否接触 | 坐标范围比较 |
| **分数系统** | 记录和展示成绩 | `useState` + localStorage |
| **生命周期** | 开始、暂停、重新开始 | 状态切换逻辑 |

> 不需要深入游戏引擎。React 的函数组件和 Hooks 已经足够完成简单游戏。

## 十一、必须记住的规则

- AI Coding 的前提是任务描述清晰、上下文准确
- 上下文越精确，AI 输出通常越可靠
- AI 生成的代码必须经过人工阅读和验证
- 复杂任务要拆分，不要一次要求修改整个项目
- Agent 负责决策，Tool 提供能力，Workflow 定义流程
- MCP 是连接 AI 与外部工具的统一协议，不是模型或数据库
- 工具权限应遵循最小权限原则
- 不应向 AI 暴露密码、Token 和私钥
- 修改后要检查 diff、运行测试并验证功能
- AI 适合辅助开发，不负责最终工程决策
- 提供完整报错比简单说"坏了"有效得多
- 安全层级：读文件 < 改文件 < 执行命令 < 部署上线

## 十二、容易混淆的概念

| 概念 A | 概念 B | 核心区别 |
|--------|--------|----------|
| **AI Coding** | **代码补全** | AI Coding 理解语义、生成完整功能；代码补全是模式匹配 |
| **Agent** | **Workflow** | Agent 自主选择步骤，灵活但需控制；Workflow 步骤固定，稳定可预期 |
| **Agent** | **Chatbot** | Agent 能调用工具执行操作；Chatbot 只能对话 |
| **Tool** | **MCP** | Tool 是具体能力；MCP 是让 AI 以统一方式调用 Tool 的协议 |
| **MCP Client** | **MCP Server** | Client 发起请求（AI 端）；Server 提供工具（资源端） |
| **自动化脚本** | **AI 自动化** | 脚本处理固定格式输入；AI 自动化处理自然语言和非结构化数据 |
| **上下文** | **提示词** | 上下文是项目文件、报错信息等事实信息；提示词是你对 AI 说的指令 |
| **AI 生成代码** | **可上线代码** | 前者是辅助产物，需要审查；后者经过验证可直接部署 |
| **读取权限** | **写入权限** | 读取只查看信息，风险低；写入会修改变更，风险高 |
| **开发环境操作** | **生产环境操作** | 开发环境允许试错；生产环境操作需审核和回滚方案 |

## 十三、本周遇到的问题与思考

### 问题 1：为什么相同指令有时会得到不同结果？

**当前理解**：LLM 的生成过程包含随机性（temperature 参数）。此外，对话历史、上下文窗口内的文件顺序、模型版本更新都会影响输出。这也解释了为什么 AI Coding 的输出需要人工检查——因为它不是确定性的。

**继续深入方向**：了解 temperature、top_p 等参数如何影响生成结果的稳定性和创造性。

### 问题 2：为什么 AI 生成的代码看起来正确，却无法运行？

**当前理解**：AI 可能不了解项目的完整依赖关系。它生成的代码可能引用了不存在的函数、使用了未安装的库版本 API、或者忽略了运行时环境差异。AI 看到的是"代码应该长什么样"，而不是"代码在真实环境中的运行结果"。

**继续深入方向**：了解 AI 模型的"幻觉"问题以及如何通过提供更精确的上下文来减少。

### 问题 3：为什么需要先让 AI 阅读项目文件？

**当前理解**：如果不提供项目文件，AI 只能"猜测"项目结构。它可能使用不存在的函数名、错误的文件路径、不匹配的数据格式。先让 AI 阅读相关文件，等于让它"了解现场情况再动工"。

**继续深入方向**：了解 RAG（检索增强生成）如何系统性地为 AI 提供项目上下文。

### 问题 4：什么任务适合 Agent，什么任务适合固定 Workflow？

**当前理解**：目标明确但路径不确定的任务适合 Agent（如"排查这个错误"）；步骤固定且频繁重复的任务适合 Workflow（如"每次 PR 时运行测试和构建"）。Agent 更灵活但更需要监督；Workflow 更稳定但不够灵活。

**继续深入方向**：了解在实际团队中如何设计 Agent 和 Workflow 的协作模式。

### 问题 5：MCP 和普通 API 有什么区别？

**当前理解**：普通 API 是应用与服务器之间的接口，通常为特定功能设计。MCP 是 AI 与工具之间的标准协议，目的是让不同工具用统一语言被 AI 调用。MCP 解决的是"AI 如何调用 N 种不同工具"的问题，普通 API 解决的是"前后端如何通信"的问题。

**继续深入方向**：了解 MCP 协议的具体规范，以及如何开发自定义 MCP Server。

### 问题 6：使用第三方 MCP Server 有哪些风险？

**当前理解**：第三方 MCP Server 可能访问敏感文件、执行未授权命令或泄露数据。它运行在本地或服务器上，权限可能超出预期。使用前需要检查：来源是否可信、请求了哪些权限、是否有代码可审查。

**继续深入方向**：了解 MCP Server 的权限模型和安全最佳实践。

### 问题 7：如何避免 AI 修改无关文件？

**当前理解**：在指令中明确指定"只修改 X 文件"、"不要修改 Y 和 Z"是有效约束。将任务拆小，每次只涉及 1-2 个文件。使用 Git 分支进行隔离，出问题可以回退。

**继续深入方向**：了解如何在 AI 工具中配置项目级别的文件访问白名单。

### 问题 8：如何判断 AI 生成的依赖是否可靠？

**当前理解**：AI 可能推荐不存在的包名、过时的版本或包含安全漏洞的依赖。安装前应该在 npm 官网查询：包是否存在、最近更新时间、下载量和维护状态。

**继续深入方向**：了解供应链安全（supply chain security）和依赖审计工具。

### 问题 9：AI 自动化是否适合直接用于生产部署？

**当前理解**：不适合。生产部署涉及数据安全、服务可用性和用户体验，错误的代价很高。AI 适合在部署前的阶段（代码生成、测试、检查），部署决策和操作应保留人工确认。

**继续深入方向**：了解 CI/CD 中 AI 的辅助角色（代码审查、测试生成）与人工决策的分界线。

### 问题 10：开发者在 AI Coding 中最重要的能力是什么？

**当前理解**：不再是"记住所有语法和 API"，而是：
1. 清楚地描述需求和约束
2. 快速判断 AI 生成的代码是否正确
3. 理解系统架构和数据流
4. 识别 AI 可能忽略的边界情况
5. 有效调试和验证代码

**继续深入方向**：随着 AI 工具进化，思考开发者角色的长期变化。

## 十五、综合实践：AI 任务工作流模拟器 + 反应速度测试

### 15.1 项目概述

配套 Demo 位于 `examples/ai-mcp-advanced-week1-demo`，包含两个模块：

- **模块一：AI 任务工作流模拟器** — 模拟 AI 自动化的分步执行过程
- **模块二：反应速度测试** — 简单交互小游戏

### 15.2 模块一：任务工作流模拟器

功能：

- 输入任务名称，选择任务类型（代码生成/调试/重构/文档）
- 自动生成四个模拟步骤：需求分析 → 上下文读取 → 任务执行 → 结果检查
- 点击步骤切换状态：待分析 → 执行中 → 待检查 → 已完成
- 删除任务
- 数据保存到 localStorage

### 15.3 模块二：反应速度测试

功能：

- 点击"开始测试"，屏幕在随机 1-5 秒后变为绿色
- 变绿后尽快点击，计算反应时间（毫秒）
- 过早点击（红色时点击）显示"太早了！"并重置
- 显示当前反应时间和最佳成绩
- 最佳成绩保存到 localStorage

### 15.4 技术说明

- React + Vite，函数组件 + Hooks
- 暗色主题，与博客风格一致
- 不调用真实 AI API，不连接 MCP Server
- 任务工作流模拟器是对 Workflow 概念的示意实现

## 核心概念速记

- AI Coding 通过理解语义辅助开发，不是简单代码补全
- 清晰的任务指令 = 背景 + 目标 + 要求 + 范围 + 限制 + 验收标准
- 上下文决定 AI 输出质量：提供真实文件、真实报错、明确范围
- Agent 自主决策路径，Workflow 按固定步骤执行，Tool 是两者的基础
- MCP 是连接 AI 与外部工具的统一协议，不是模型也不是数据库
- 安全层级：读文件 < 改文件 < 执行命令 < 部署上线
- AI 生成代码必须人工检查；依赖需要核实；敏感信息不能暴露
- 调试时提供完整报错，不要转述；一次只改 1-2 处
- 开发者在 AI 时代最重要的能力：描述需求、判断代码、理解架构
