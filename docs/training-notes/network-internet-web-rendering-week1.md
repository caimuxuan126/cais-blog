## 一、前置知识

### 1.1 网络解决了什么问题

计算机最初是单机工作的。随着需求增长，三个核心问题浮现：

| 需求 | 场景 |
|------|------|
| **数据共享** | 将文件从 A 电脑传输到 B 电脑 |
| **资源共享** | 多台电脑共用一台打印机 |
| **通信** | 即时消息、联机协作 |

网络的本质：**让计算机之间能够互相通信**。

### 1.2 主机、客户端、服务器

| 概念 | 定义 | 说明 |
|------|------|------|
| **主机 (Host)** | 任何连入网络的设备 | 笔记本、手机、云服务器都是主机 |
| **客户端 (Client)** | 主动发起请求的一方 | 浏览器就是典型客户端 |
| **服务器 (Server)** | 被动等待请求、提供服务的一方 | 是一个**角色**，不是特定硬件；任何主机安装服务器软件后都可以充当服务器 |

### 1.3 局域网、广域网、互联网

```
两台直连 → 局域网 (LAN) → 广域网 (WAN) → 互联网 (Internet)
```

| 类型 | 范围 | 示例 |
|------|------|------|
| **局域网 (LAN)** | 小范围（同一建筑内） | 家庭网络、公司内网、学校机房 |
| **广域网 (WAN)** | 跨城市、跨国家 | 公司北京与上海办公室之间的专线 |
| **互联网 (Internet)** | 全球范围 | 将全世界无数 LAN 和 WAN 互联起来的"网络的网络" |

> **关键理解**：Internet = Inter + net（网络之间的网络）。互联网不是一个单一网络，而是无数网络的互联体。

### 1.4 IP、端口、域名

三者解决的问题逐层递进：

```
定位主机              定位主机上的程序            让人能记住
    ↓                       ↓                       ↓
  IP 地址                 端口号                   域名
192.168.1.1               :3000               google.com
```

| 概念 | 解决的问题 | 类比 |
|------|-----------|------|
| **IP 地址** | 在网络中定位一台主机 | 楼宇的门牌号 |
| **端口号** | 在一台主机上定位具体程序 | 楼内某个房间号 |
| **域名** | 用可读名称代替数字 IP | 给门牌号起一个好记的名字 |

### 1.5 localhost 与 127.0.0.1

- `localhost` 是一个特殊的主机名，始终指向**本机**
- `127.0.0.1` 是 IPv4 的**回环地址 (loopback address)**
- 访问 `localhost:3000` 等价于访问 `127.0.0.1:3000`，数据不会离开本机网卡

> 开发中运行 `npm run dev` 后浏览器打开的 `localhost:3000` 就是在访问本机上运行的开发服务器。

## 二、计算机网络分层

### 2.1 为什么需要分层

网络通信涉及硬件信号、数据纠错、路径选择、应用协议等众多问题。如果所有逻辑混在一起，开发和维护将变得极其困难。

**分层思想**：将复杂任务拆成多个独立的层，每层只关注自己的职责，向上层提供服务，使用下层提供的服务。类似软件工程中的模块化设计。

### 2.2 OSI 七层模型 vs TCP/IP 四层模型

```
OSI 七层（理论）           TCP/IP 四层（实际）        常见协议
───────────────────────────────────────────────────────────────
7. 应用层                                             HTTP, HTTPS
6. 表示层  ────┐                                    FTP, SMTP
5. 会话层  ────┼──→  应用层                          DNS, SSH
              │
4. 传输层  ────┼──→  传输层                          TCP, UDP
              │
3. 网络层  ────┼──→  网络层                          IP, ICMP
              │
2. 链路层  ────┼──→  网络接口层                       Ethernet, Wi-Fi
1. 物理层  ────┘                                      光纤, 无线电
```

### 2.3 各层职责速查

| 层 | 核心问题 | 关键概念 | 数据单元 |
|----|---------|---------|----------|
| **应用层** | 数据怎么用？ | HTTP, DNS, SMTP, FTP | 消息 (Message) |
| **传输层** | 数据怎么可靠/快速传输？ | TCP, UDP, 端口号 | 段 (Segment) |
| **网络层** | 数据走哪条路？ | IP, 路由, 路由器 | 包 (Packet) |
| **链路层+物理层** | 信号怎么发送？ | MAC 地址, 帧, 交换机 | 帧 (Frame) / 比特 (Bit) |

### 2.4 数据在各层之间的传递

```
发送方                                    接收方
──────                                   ──────
应用层: [HTTP 数据]                       应用层: [HTTP 数据]
           ↓ 封装                               ↑ 解封装
传输层: [TCP 头 | HTTP 数据]              传输层: [TCP 头 | HTTP 数据]
           ↓                                       ↑
网络层: [IP 头 | TCP 头 | HTTP 数据]       网络层: [IP 头 | TCP 头 | HTTP 数据]
           ↓                                       ↑
链路层: [帧头 | IP 头 | TCP 头 | HTTP | 帧尾] 链路层: [帧头 | ... | 帧尾]
           ↓                                       ↑
          ═══════ 物理介质传输 ═══════
```

每一层在数据前面加上自己的头部信息（封装），接收方逐层去掉头部（解封装），最终应用层拿到原始数据。

## 三、传输层基础：TCP 与 UDP

### 3.1 TCP 的特点

- **面向连接**：通信前需要建立连接（三次握手）
- **可靠传输**：保证数据按序、完整到达，有确认重传机制
- **流量控制与拥塞控制**：根据网络状况调整发送速率
- **全双工**：双方可以同时发送和接收数据

### 3.2 UDP 的特点

- **无连接**：不需要建立连接，直接发送数据
- **不保证可靠**：可能丢包、乱序，没有确认机制
- **速度快**：没有连接建立和确认重传的开销
- **支持广播和多播**：可以同时向多个目标发送

### 3.3 TCP vs UDP 对比

| 对比维度 | TCP | UDP |
|----------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 保证送达，不丢不乱 | 不保证 |
| 速度 | 较慢（确认机制开销） | 快 |
| 适用场景 | 网页、文件下载、邮件 | 视频直播、语音通话、在线游戏 |
| 类比 | 打电话——先拨通，确认对方在听再说话 | 发短信——发送即完成，不确认对方是否已读 |

> **选择原则**：没有绝对的优劣，只有场景的适配。需要可靠传输用 TCP，需要低延迟用 UDP。

### 3.4 TCP 三次握手（建立连接）

```
客户端                                    服务器
  │                                         │
  │──── SYN (seq=x) ────────────────────→  │  第1次：客户端请求建立连接
  │                                         │
  │←── SYN+ACK (seq=y, ack=x+1) ────────  │  第2次：服务器同意并确认
  │                                         │
  │──── ACK (ack=y+1) ─────────────────→  │  第3次：客户端确认收到
  │                                         │
  │═══════════ TCP 连接已建立 ══════════════│
```

**为什么是三次？** 防止已失效的连接请求到达服务器导致资源浪费。如果只用两次，客户端发送请求后掉线，服务器会一直保持无效连接。

### 3.5 TCP 四次挥手（断开连接）

```
客户端                                    服务器
  │                                         │
  │──── FIN (seq=u) ───────────────────→  │  第1次：客户端说"我没有数据要发了"
  │                                         │
  │←── ACK (ack=u+1) ──────────────────  │  第2次：服务器说"知道了"
  │                                         │ (服务器可能还有数据要发)
  │←── FIN (seq=v, ack=u+1) ────────────  │  第3次：服务器说"我也发完了"
  │                                         │
  │──── ACK (ack=v+1) ─────────────────→  │  第4次：客户端说"好的，再见"
  │                                         │
  │═══════════ TCP 连接已关闭 ══════════════│
```

**为什么是四次？** TCP 是全双工的，双方可以独立决定何时停止发送。一方完成发送不代表另一方也完成，因此需要各自发送 FIN 和确认。

### 3.6 常见使用场景

| 协议 | 典型应用 |
|------|---------|
| **TCP** | HTTP/HTTPS、FTP、SMTP、SSH、WebSocket |
| **UDP** | DNS 查询、VoIP 语音、视频会议、在线游戏、DHCP |

## 四、应用层基础：HTTP、HTTPS 与 DNS

### 4.1 HTTP 是什么

**HTTP (HyperText Transfer Protocol)** 是 Web 通信的基础协议，定义了客户端（浏览器）和服务器之间如何请求和传输数据。

核心特点：
- **请求-响应模型**：客户端发起请求，服务器返回响应
- **无状态**：每次请求独立，服务器不记住上一次交互的内容（通过 Cookie/Session/Token 解决）
- **基于 TCP**：HTTP/1.x 和 HTTP/2 运行在 TCP 之上

### 4.2 HTTP 请求与响应的基本结构

**请求报文：**

```
GET /index.html HTTP/1.1          ← 请求行：方法 + 路径 + 协议版本
Host: www.example.com             ← 请求头
User-Agent: Mozilla/5.0
Accept: text/html
                                  ← 空行（头与体之间）
(请求体 — GET 请求通常为空)         ← 请求体
```

**响应报文：**

```
HTTP/1.1 200 OK                   ← 状态行：协议版本 + 状态码 + 原因短语
Content-Type: text/html           ← 响应头
Content-Length: 1234
Set-Cookie: session=abc123
                                  ← 空行
<html>...</html>                   ← 响应体
```

### 4.3 常见 HTTP 方法

| 方法 | 含义 | 幂等性 | 典型用途 |
|------|------|--------|----------|
| **GET** | 获取资源 | ✅ 是 | 访问网页、查询数据 |
| **POST** | 创建资源 / 提交数据 | ❌ 否 | 表单提交、登录、上传 |
| **PUT** | 完整更新资源 | ✅ 是 | 覆盖已有资源 |
| **DELETE** | 删除资源 | ✅ 是 | 删除数据 |

### 4.4 常见 HTTP 状态码

| 状态码 | 含义 | 说明 |
|--------|------|------|
| **200** | OK | 请求成功 |
| **301** | 永久重定向 | 资源已永久移动到新 URL |
| **302** | 临时重定向 | 资源暂时移动到新 URL |
| **304** | 未修改 | 使用缓存版本（与缓存机制配合） |
| **400** | 请求错误 | 客户端请求有语法错误 |
| **401** | 未授权 | 需要认证 |
| **403** | 禁止访问 | 服务器理解请求但拒绝执行 |
| **404** | 未找到 | 请求的资源不存在 |
| **500** | 服务器内部错误 | 服务器代码出错 |

> **记忆口诀**：2xx 成功，3xx 重定向，4xx 客户端错误，5xx 服务端错误。

### 4.5 HTTPS 与 HTTP 的区别

```
HTTP:   [浏览器] ──── 明文传输 ──── [服务器]    ← 可被窃听、篡改
HTTPS:  [浏览器] ──── 加密传输 ──── [服务器]    ← 防窃听、防篡改、验证身份
                       ↑
                    TLS/SSL 层
```

| 对比 | HTTP | HTTPS |
|------|------|-------|
| 加密 | 无，明文 | TLS/SSL 加密 |
| 默认端口 | 80 | 443 |
| 安全性 | 低 | 高 |
| 性能 | 略快 | 略慢（加解密和握手开销） |

> **HTTPS = HTTP over TLS**。TLS 层位于 TCP 之上、HTTP 之下，对 HTTP 数据进行加密。

**TLS 握手简化过程：**
1. 客户端发起请求，附上支持的加密算法列表
2. 服务器返回数字证书（包含公钥）
3. 客户端验证证书合法性
4. 双方协商生成对称密钥，后续通信使用该密钥加密

### 4.6 DNS 的作用与解析流程

**DNS (Domain Name System)** 是互联网的"电话本"——将人类可读的域名翻译为机器可读的 IP 地址。

```
输入 google.com
      │
  ┌───▼──── 1. 浏览器 DNS 缓存     命中？→ 返回 IP
  │   (chrome://net-internals/#dns)
  ├───▼──── 2. 操作系统缓存        命中？→ 返回 IP
  │   (hosts 文件 + OS DNS 缓存)
  ├───▼──── 3. 本地 DNS 服务器     命中？→ 返回 IP
  │   (路由器 / ISP 提供的 DNS)
  ├───▼──── 4. 根 DNS 服务器
  │   "去问 .com 的服务器"
  ├───▼──── 5. 顶级域 DNS (.com)
  │   "去问 google.com 的服务器"
  └───▼──── 6. 权威 DNS (google.com)
      "IP 地址是 142.250.80.46"
              │
              ▼
      结果逐级缓存，返回给浏览器
```

> `nslookup google.com` 或 `dig google.com` 可以手动查看 DNS 解析结果。

## 五、登录状态相关概念

### 5.1 HTTP 的无状态性

HTTP 协议本身不保存客户端的状态——每次请求都是独立的，服务器无法"记住"你是谁。但对于现代 Web 应用，保持登录状态是基本需求。

### 5.2 三种状态保持方案

| 方案 | 存储位置 | 工作原理 | 优缺点 |
|------|---------|---------|--------|
| **Cookie** | 浏览器端 | 服务器通过 `Set-Cookie` 响应头让浏览器存储一段数据，后续请求自动携带 | 容量小(4KB)、每次请求都发送、可设置 HttpOnly 防 JS 读取 |
| **Session** | 服务器端 | 服务器保存用户数据，只给浏览器一个 `session_id`（通常存在 Cookie 中） | 服务器压力大、分布式环境需共享 Session |
| **Token (JWT)** | 浏览器端 | 服务器签发一个包含用户信息的加密令牌，浏览器在请求头中携带 | 无状态、易扩展、需防 XSS 窃取 |

### 5.3 Cookie、Session、Token 对比

| 维度 | Cookie | Session | Token (JWT) |
|------|--------|---------|-------------|
| 数据存储 | 浏览器 | 服务器 | 浏览器 |
| 安全性 | 可设置 HttpOnly | 较高（数据在服务端） | 依赖 HTTPS + 存储方式 |
| 扩展性 | 单机 OK | 分布式需额外处理 | 天然支持分布式 |
| 典型场景 | 记住用户偏好 | 传统 Web 登录 | SPA、移动端、API 鉴权 |

> **生产环境常见实践**：JWT 存储在 HttpOnly Cookie 中，兼顾安全性和扩展性。

## 六、互联网基本认识

### 6.1 互联网不是单一系统

互联网是全球无数网络通过标准协议（TCP/IP）互联形成的开放网络。它不是由单一组织拥有或控制的。

### 6.2 核心角色

| 角色 | 功能 | 类比 |
|------|------|------|
| **浏览器** | 发送请求、渲染页面 | 用户的门户 |
| **服务器** | 存储和处理数据、响应请求 | 提供服务的后端 |
| **DNS** | 域名→IP 的翻译 | 电话本 |
| **CDN** | 将静态资源缓存到离用户近的节点 | 全国连锁便利店（就近取货） |
| **路由器** | 在不同网络之间转发数据包 | 快递中转站 |
| **网关** | 连接不同类型的网络，做协议转换 | 翻译官 |

### 6.3 CDN 为什么能加速

- 将静态资源（JS、CSS、图片、视频）分发到全球各地的边缘节点
- 用户请求被引导到**地理上最近**的节点，减少网络延迟
- 减轻源站压力，提高并发能力

### 6.4 前端、后端、服务器、数据库的关系

```
用户浏览器（前端）
    │  HTTP 请求
    ▼
Web 服务器 (Nginx / Apache)
    │  转发动态请求
    ▼
应用服务器 (Node.js / Java / Python)
    │  读写数据
    ▼
数据库 (MySQL / PostgreSQL / MongoDB)
    │
    ▼ 返回数据 → 应用服务器处理 → 返回 HTML/JSON → 浏览器渲染
```

| 角色 | 职责 |
|------|------|
| **前端** | 负责界面展示和用户交互（HTML/CSS/JS） |
| **后端** | 负责业务逻辑和数据处理 |
| **服务器** | 运行前端和后端代码的硬件或软件 |
| **数据库** | 持久化存储数据 |

## 七、从输入 URL 到页面展示

当用户在浏览器输入 `https://example.com/page` 并按下回车后，经历了以下完整流程：

### 第一阶段：网络请求

| 步骤 | 操作 | 说明 |
|------|------|------|
| **1. 解析 URL** | 浏览器解析协议(https)、主机名(example.com)、路径(/page) | 检查 URL 合法性 |
| **2. 查询缓存** | 浏览器缓存 → Service Worker → 强缓存/协商缓存 | 命中缓存则直接返回，跳过后续网络步骤 |
| **3. DNS 解析** | 域名 → IP 地址 | 见 [4.6 DNS 解析流程](#46-dns-的作用与解析流程) |
| **4. TCP 连接** | 三次握手 | 见 [3.4 TCP 三次握手](#34-tcp-三次握手建立连接) |
| **5. TLS 握手** | HTTPS 的加密握手 | 验证证书 + 协商密钥 |
| **6. 发送 HTTP 请求** | `GET /page HTTP/1.1` | 包含请求头（Host, User-Agent, Cookie 等） |
| **7. 服务器处理** | 后端处理请求（查数据库、渲染模板等） | 生成 HTML 响应 |
| **8. 返回 HTTP 响应** | `HTTP/1.1 200 OK` | 包含状态码、响应头(Content-Type, Cache-Control 等)、HTML 内容 |

### 第二阶段：浏览器渲染

| 步骤 | 操作 | 说明 |
|------|------|------|
| **9. 接收 HTML** | 浏览器开始接收并解析 HTML | 边下载边解析 |
| **10. 解析 HTML** | 构建 DOM 树 | 遇到 `<script>` 可能阻塞解析 |
| **11. 加载子资源** | 解析到 CSS/JS/图片等 | 发起额外 HTTP 请求 |
| **12. 构建 CSSOM** | 解析 CSS，构建 CSS 对象模型 | CSS 解析不阻塞 DOM 构建，但阻塞渲染 |
| **13. 生成 Render Tree** | DOM + CSSOM → 渲染树 | 不可见元素(display:none)不进入渲染树 |
| **14. Layout (布局)** | 计算每个节点的精确位置和大小 | 也称为回流(Reflow) |
| **15. Paint (绘制)** | 生成绘制指令 | 填充像素 |
| **16. Composite (合成)** | 图层合并，提交 GPU | 最终显示在屏幕上 |
| **17. 页面展示** | 用户看到完整页面 | 后续交互触发增量更新 |

### 整体流程图

```
                   【网络阶段】
  URL 输入 → 缓存检查 → DNS 解析 → TCP 连接 → TLS 握手
                                                 ↓
                                            HTTP 请求
                                                 ↓
                                            服务器处理
                                                 ↓
                                            HTTP 响应
                                                 ↓
                   【渲染阶段】
  ←────────── 接收 HTML ──────────→
        解析 HTML          下载子资源(CSS/JS/Img)
           ↓                     ↓
       构建 DOM             构建 CSSOM
           ↓                     ↓
           └────── 合并 ─────────┘
                      ↓
                 Render Tree
                      ↓
              Layout (布局/回流)
                      ↓
              Paint (绘制)
                      ↓
              Composite (合成)
                      ↓
              ═══ 页面展示 ═══
```

## 八、Web 渲染原理

### 8.1 三种语言的角色

| 技术 | 角色 | 类比 |
|------|------|------|
| **HTML** | 定义页面结构与内容 | 建筑的骨架 |
| **CSS** | 定义页面样式与布局 | 建筑的装修 |
| **JavaScript** | 定义页面交互与动态行为 | 建筑的电路和管道系统 |

### 8.2 关键概念

| 概念 | 全称 | 定义 |
|------|------|------|
| **DOM** | Document Object Model | HTML 解析后生成的树状结构，表示文档的内容和结构 |
| **CSSOM** | CSS Object Model | CSS 解析后生成的树状结构，表示样式规则 |
| **Render Tree** | 渲染树 | DOM + CSSOM 合并后的树，只包含可见元素 |
| **Layout** | 布局/回流 | 计算每个可见元素的精确位置和尺寸 |
| **Paint** | 绘制 | 将布局结果转换为像素填充指令 |
| **Composite** | 合成 | 将不同图层合并为一帧，交给 GPU 输出到屏幕 |

### 8.3 渲染流程

```
HTML ──解析──→ DOM Tree ──┐
                          ├──→ Render Tree ──→ Layout ──→ Paint ──→ Composite
CSS  ──解析──→ CSSOM Tree ─┘
```

### 8.4 回流 (Reflow) 与重绘 (Repaint)

| 类型 | 触发条件 | 代价 | 示例属性 |
|------|---------|------|---------|
| **回流 Reflow** | 几何属性改变 | 🔴 高 | width, height, margin, padding, left, top, font-size |
| **重绘 Repaint** | 外观属性改变 | 🟡 中 | color, background, box-shadow, border-color |
| **仅合成** | transform/opacity 改变 | 🟢 低 | transform, opacity |

```
修改 width → 回流 → 重绘 → 合成（最慢）
修改 color → 重绘 → 合成
修改 transform → 仅合成（最快）
```

> **工程建议**：动画使用 `transform` 和 `opacity`，避免使用 `left`/`top`/`width` 做动画。

### 8.5 为什么频繁操作 DOM 影响性能

- 每次修改 DOM 或 CSS 都可能触发回流或重绘
- 频繁的 Layout 计算是高开销操作
- **优化策略**：
  - 批量修改 DOM（使用 `DocumentFragment` 或先 `display:none`）
  - 使用 CSS class 切换代替逐个属性修改
  - 动画元素使用 `position: fixed/absolute` 脱离文档流
  - React/Vue 的 Virtual DOM 本质上就是通过 diff 减少实际 DOM 操作

### 8.6 script 标签的阻塞行为

| 标签写法 | 行为 | 使用场景 |
|----------|------|----------|
| `<script src="a.js">` | 阻塞 HTML 解析，下载并立即执行 | 需要立即执行的脚本 |
| `<script async src="a.js">` | 异步下载，下载完立即执行，不保证顺序 | 独立脚本（如统计、广告） |
| `<script defer src="a.js">` | 异步下载，等 HTML 解析完毕再按顺序执行 | 依赖 DOM 的脚本 |

```
普通 script:  HTML解析 ██░░░░░░░░██████  (中间阻塞等待下载+执行)
async:        HTML解析 ██░░░░██░░░░████  (下载完就执行)
defer:        HTML解析 ████████████░░░░  (解析完才执行)
```

## 九、浏览器性能与调试

### 9.1 Chrome DevTools Network 面板

**打开方式**：F12 → Network 标签 → 刷新页面

**关键指标：**

| 指标 | 含义 | 参考值 |
|------|------|--------|
| **Queueing** | 请求排队等待时间 | 越小越好 |
| **DNS Lookup** | DNS 解析耗时 | < 50ms |
| **Initial connection** | TCP 握手耗时 | < 100ms |
| **SSL** | TLS 握手耗时 | < 100ms |
| **TTFB** | 等待服务器响应第一个字节 | < 200ms（理想） |
| **Content Download** | 下载响应体耗时 | 取决于资源大小 |

> **TTFB (Time To First Byte)** 是衡量服务器响应速度的关键指标。TTFB 过大说明服务器处理慢或网络延迟高。

**瀑布图 (Waterfall)**：每条横条代表一个请求的时间线，颜色分段对应上述各个阶段。

### 9.2 Performance 面板

**用途**：分析页面运行时性能，找出瓶颈。

**操作流程**：Performance → 点击录制按钮 → 操作页面 → 停止 → 分析结果

**关键色块：**
- 🔵 **蓝色 (Loading)**：网络请求和 HTML 解析
- 🟡 **黄色 (Scripting)**：JavaScript 执行
- 🟣 **紫色 (Rendering)**：样式计算和布局
- 🟢 **绿色 (Painting)**：绘制

**关注指标：**
- **FCP (First Contentful Paint)**：首次内容绘制时间
- **LCP (Largest Contentful Paint)**：最大内容绘制时间
- **长任务 (Long Tasks)**：超过 50ms 的任务，可能造成卡顿

### 9.3 判断性能瓶颈

| 问题表现 | 可能原因 | 排查方向 |
|----------|---------|----------|
| 白屏时间长 | TTFB 高或关键资源加载慢 | Network 面板看 TTFB 和关键请求耗时 |
| 页面卡顿 | 频繁回流或 JS 长任务 | Performance 面板找长任务、检查 Layout 事件 |
| 动画不流畅 | 触发了回流而非仅合成 | 检查动画属性是否使用了 left/top 等 |
| 首屏慢 | 资源过多或未优化 | 检查资源数量和大小、考虑懒加载和代码分割 |

## 十一、常见问题与容易混淆的概念

| 问题 | 解答 |
|------|------|
| **TCP 和 HTTP 是什么关系？** | HTTP 是应用层协议，依赖传输层的 TCP 进行可靠数据传输。HTTP 负责"说什么"，TCP 负责"怎么送" |
| **IP 地址和 MAC 地址有什么区别？** | IP 是逻辑地址（网络层），用于跨网络定位；MAC 是物理地址（链路层），用于同一网络内的设备识别 |
| **Session 和 Token 用哪个？** | 传统服务端渲染用 Session；SPA/移动端/微服务用 Token (JWT) |
| **回流和重绘哪个更消耗性能？** | 回流远大于重绘。回流触发几何重新计算，重绘只需重新填充像素 |
| **HTTPS 比 HTTP 慢多少？** | 首次连接有 TLS 握手开销（几十到上百毫秒），后续请求几乎无差异（HTTP/2 下甚至可能更快） |
| **localhost 能在外网访问吗？** | 不能。localhost 仅指向本机，外网无法访问。需要部署到公网 IP 或使用内网穿透工具 |

## 核心概念速记

| 概念 | 一句话 |
|------|--------|
| **网络分层** | 把复杂通信拆成多层，每层只做自己的事 |
| **TCP** | 可靠、面向连接、三次握手、四次挥手 |
| **UDP** | 快速、无连接、不保证送达 |
| **HTTP** | 请求-响应模型、无状态、基于 TCP |
| **HTTPS** | HTTP + TLS 加密 |
| **DNS** | 域名 → IP 的翻译系统 |
| **DOM** | HTML 的树状对象表示 |
| **CSSOM** | CSS 的树状对象表示 |
| **Render Tree** | DOM + CSSOM 合并后的可见元素树 |
| **回流** | 几何属性改变导致重新布局（高开销） |
| **重绘** | 外观属性改变导致重新填充像素（中开销） |
| **合成** | GPU 图层合并（低开销，动画首选） |
